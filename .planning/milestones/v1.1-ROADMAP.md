# Milestone v1.1: Search Depth and Scale

**Status:** ✅ SHIPPED 2026-03-22
**Phases:** 4-6
**Total Plans:** 9

## Overview

`tree-sitter-mcp` v1.1 strengthened semantic retrieval in three layers: persistent indexed state for faster repeated queries, federated multi-workspace search with precise attribution and narrowing, and relationship-aware retrieval that lets agents inspect direct impact around a symbol. The milestone kept the MCP surface read-only and intentionally deferred HTTP transport until search quality, scale, and depth were proven first.

## Phases

### Phase 4: Persistent Indexing and Query Freshness

**Goal**: Introduce reusable semantic state so repeated queries become faster without returning stale answers or hiding freshness details from the caller.
**Depends on**: Phase 3
**Plans**: 3 plans

Plans:

- [x] 04-01: Design persistent semantic state, storage boundaries, and invalidation rules.
- [x] 04-02: Implement index/cache build, reuse, and targeted refresh paths for repeated queries.
- [x] 04-03: Surface freshness diagnostics and add regression coverage for stale-result prevention.

**Details:**
Built versioned persistent-index storage, workspace fingerprinting, and coordinator-managed freshness state; then routed symbol/definition/reference flows through indexed records with targeted refresh and degraded-file protection; and finally exposed explicit freshness/index metadata through bootstrap, capabilities, health, and search payloads with restart-reuse and degraded-search coverage.

### Phase 5: Multi-Workspace Search and Result Quality

**Goal**: Let agents search across multiple workspace roots while keeping large result sets understandable, attributable, and filterable.
**Depends on**: Phase 4
**Plans**: 3 plans

Plans:

- [x] 05-01: Extend workspace configuration and discovery to support multiple active roots.
- [x] 05-02: Make search pipelines and result payloads workspace-aware with stronger narrowing controls.
- [x] 05-03: Validate search quality on larger or federated repositories and refine result shaping.

**Details:**
Expanded `set_workspace` and the semantic index to multiple ordered roots, carried `workspaceRoot` through every result and follow-up lookup surface, added centralized workspace/language/path/kind narrowing plus deterministic federated ranking, and exposed per-workspace breakdown metadata with dedicated multi-root regression coverage.

### Phase 6: Relationship Views and Impact Discovery

**Goal**: Build relationship-aware retrieval on top of the stronger search foundation so agents can inspect direct links and likely impact around a symbol.
**Depends on**: Phase 5
**Plans**: 3 plans

Plans:

- [x] 06-01: Model relationship queries and schemas using existing definition/reference primitives.
- [x] 06-02: Implement relationship traversal, filtering, pagination, and workspace-aware payload shaping.
- [x] 06-03: Validate relationship and impact workflows against realistic multi-workspace repositories.

**Details:**
Defined relationship schemas and filter normalization, implemented indexed incoming/outgoing relationship traversal plus the read-only `get_relationship_view` MCP tool, and finished with single-root and federated E2E validation that proved workspace attribution, pagination, hop expansion, and refreshed/degraded freshness behavior.

## Milestone Summary

**Decimal Phases:**

- None.

**Key Decisions:**

- Keep v1.1 focused on search improvements before any transport expansion.
- Pair persistent indexing and multi-workspace search because both require shared workspace-state and freshness tracking primitives.
- Start relationship retrieval with direct semantic links and one extra impact hop instead of attempting whole-program graphs immediately.
- Preserve one read-only MCP surface by reusing shared definition/reference/index backends rather than introducing transport-specific or mutation-oriented forks.
- Prefer same-file and same-workspace resolution before global fallback when relationship traversal encounters duplicate names across roots.

**Issues Resolved:**

- Added persistent-index freshness diagnostics so indexed search never silently serves stale answers after workspace changes.
- Kept duplicate relative paths isolated across workspaces by using `{ workspaceRoot, relativePath }` as record identity.
- Fixed federated relationship edge disambiguation so duplicate names in different roots do not bleed across relationship views.
- Preserved refreshed/degraded freshness metadata through multi-step relationship resolution workflows.

**Issues Deferred:**

- Streamable HTTP transport and alternate connection modes.
- Deeper relationship analysis beyond direct edges plus one extra impact hop.
- Safe write/refactor workflows on top of the read-only retrieval surface.
- Broader grammar/language coverage beyond the current builtins.

**Technical Debt Incurred:**

- Summary frontmatter still does not emit machine-readable one-liners, so milestone archival automation underreports accomplishments unless manually curated.
- No dedicated v1.1 milestone audit artifact was created before archival.
- Relationship retrieval is intentionally neighborhood-scoped (`maxDepth` 1..2) and may need a separate planning cycle before deeper graph workflows are introduced.
